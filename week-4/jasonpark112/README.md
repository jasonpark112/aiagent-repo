## 2. 인덱싱 전략

2025년, 2026년 PDF를 각각 마크다운으로 변환한 후 아래 전략으로 청킹했다.

### Parent-Child 청킹 구조

각 child chunk의 메타데이터에 `source_year`, `parent_id`, `parent_content`를 저장하여 검색 시 child로 찾고, LLM에는 parent_content를 전달하는 Parent-Child 구조를 채택했다.

```python
metadata = {
    "source_year": "2025",       # 년도 필터링 기준
    "parent_id": "2025_p0",      # 중복 parent 제거용
    "parent_content": "...",     # LLM에 전달할 넓은 컨텍스트
}
```

### 청크 크기 실험

초기 설정(c300_p1500)에서 정확도가 낮아 청크 크기를 확대(c1000_p3000)하여 재실험했다.

| 구분 | child chunk | parent chunk | 비고 |
|------|------------|--------------|------|
| 초기 (c300_p1500) | 300자, overlap 50자 | 1500자, overlap 100자 | HTML 표 행 분절 문제 발생 |
| 개선 (c1000_p3000) | 1000자, overlap 80자 | 3000자, overlap 150자 | HTML 표 행 단위 의미 보존 |

## 이론 과제

### Q1. Hybrid Search란 무엇이며 왜 효과적인가?
Hybrid Search는 키워드 기반 검색(BM25/TF-IDF)과 의미론적 벡터 검색(Dense Retrieval)을 결합한 검색 방식입니다.

| 방식 | 원리 | 강점 | 약점 |
|------|------|------|------|
| **키워드 검색 (Sparse)** | 단어 빈도 기반 매칭 | 정확한 용어, 고유명사에 강함 | 의미적 유사성 파악 불가 |
| **벡터 검색 (Dense)** | 임베딩 공간 유사도 | 의미·맥락 이해에 강함 | 특정 단어 매칭에 약함 |
| **Hybrid** | 두 점수를 RRF 등으로 결합 | 둘의 장점을 모두 활용 | 가중치 튜닝 필요 |

두 점수를 합칠 때는 주로 **RRF(Reciprocal Rank Fusion)** 를 사용합니다.

```
score = Σ weight_i / (rank_i + k)   # k=60 default
```

### Q2. Re-ranking은 왜 필요하며 Cross-encoder와 Bi-encoder의 차이는?
1차 검색(Retrieval)은 속도를 위해 근사치로 동작하기 때문에, 상위 결과가 항상 관련성 높은 문서는 아닐 수 있습니다. Re-ranking은 1차로 뽑은 후보군(Top-K)을 더 정밀하게 재정렬하는 2단계 과정입니다.

| | **Bi-encoder** | **Cross-encoder** |
|---|---|---|
| **동작** | 쿼리·문서를 각각 인코딩 후 유사도 계산 | 쿼리+문서를 함께 넣어 관련성 점수 출력 |
| **속도** | 빠름 (사전 임베딩 가능) | 느림 (쌍마다 추론 필요) |
| **정확도** | 보통 | 높음 |
| **역할** | 1차 검색 (수백만 문서 대상) | Re-ranking (수십 후보 대상) |

### Q3. 메타데이터 필터링과 컨텍스트 압축의 차이 및 활용 상황은?

#### 메타데이터 필터링
검색 이전에 조건을 걸어 대상 문서 자체를 줄이는 방식입니다.

```python
def filter_by_year(docs, source_year):
    filtered = [d for d in docs if d.metadata.get("source_year", "") == source_year]
    return filtered if filtered else docs
```

**활용 상황:** 날짜, 카테고리, 출처, 작성자 등으로 범위를 좁힐 수 있을 때 사용합니다.

#### 컨텍스트 압축 (Contextual Compression)
문서를 가져온 이후에 쿼리와 관련 없는 내용을 제거해 핵심만 남기는 방식입니다.

```
[원본 청크: 500 tokens]  →  [압축 후: 120 tokens, 핵심 내용만]
```

**활용 상황:** 청크가 크거나 노이즈가 많아 LLM 컨텍스트 낭비가 심할 때 사용합니다.

## 실습 과제 예측 및 결과 비교

### 1. 년도 혼동이 가장 많이 발생할 질문 유형

**예측**
2025년과 2026년 문서 구조가 유사한 항목, 특히 본인부담률 표 형식이 동일한 항목에서 년도 혼동이 많이 발생할 것으로 예측했다. 추나요법, 정신질환 외래처럼 두 문서에 동일한 키워드가 반복되는 항목이 취약할 것으로 봤다.

**실제 결과**
예측이 맞았다. q15(복잡추나 2026)에서 2025 청크만 반환되는 년도 혼선이 발생했다. BM25가 키워드 매칭으로 두 연도의 유사 문서를 동시에 끌어오는 경우가 많았고, Hybrid Only 방식에서 년도 혼선이 가장 빈번했다.

---

### 2. Hybrid Search가 가장 효과를 볼 질문 유형

**예측**
"복잡추나", "선택의료급여기관", "장기지속형 주사제"처럼 도메인 고유 용어가 포함된 질문에서 BM25의 키워드 매칭이 벡터 검색을 보완할 것으로 예측했다.

**실제 결과**
부분적으로 맞았다. Hybrid + Re-ranking (c1000_p3000, no filter)가 70%로 가장 높은 정답률을 보였지만, Hybrid Only 단독은 30%로 Basic RAG(40%)보다 낮았다. Re-ranking 및 청크 크기와의 조합에서 효과가 나타났다.

---

### 3. Re-ranking이 결과를 개선 또는 악화시키는 경우

**예측**
Re-ranking이 전반적으로 정답률을 높일 것으로 예측했다. 다만 후보 풀이 적거나 관련 청크 자체가 없을 경우 오히려 노이즈 청크를 상위로 올릴 수 있다고 봤다.

**실제 결과**
예측이 맞았다. Hybrid Only(30%) → Hybrid + Re-ranking no_filter(45%)로 15%p 상승해 Re-ranking 효과는 유효했다. 반면 년도 필터링 후 후보 풀이 줄어든 상태에서 Re-ranking을 적용한 year_filter 버전은 30%로 악화됐다(q03, q14, q20).

---

### 4. 메타데이터 필터링의 년도 혼동 해결 효과

**예측**
메타데이터 필터링이 년도 혼동을 완전히 제거할 수 있을 것으로 예측했다. 다만 필터링으로 후보 풀이 좁아져 관련 청크 자체가 탈락하는 부작용이 있을 수 있다고 봤다.

**실제 결과**
예측이 정확했다. year_filter 버전은 년도 혼선 0%를 달성했지만, 정답률은 30%로 가장 낮았다. 필터가 후보 풀을 좁혀 관련 청크가 탈락하는 역효과가 실제로 발생했다.

## Step 1: Basic RAG

### 파이프라인 구성

```
질문 → FAISS 벡터 검색 (Top-3) → parent_content 컨텍스트 구성 → LLM → 답변
```

- **Retriever**: `vectorstore.similarity_search(question, k=3)`
- **LLM**: `gpt-4o-mini`, temperature=0
- **프롬프트**: 컨텍스트에 `[출처 년도: YYYY]`를 명시하여 LLM이 년도를 인식하도록 구성

```
아래 컨텍스트를 바탕으로 질문에 답하세요.
각 컨텍스트에는 출처 년도가 표시되어 있습니다. 질문이 특정 년도를 묻는 경우 해당 년도의 정보만 사용하세요.
컨텍스트에 없는 내용은 "정보를 찾을 수 없습니다"라고 답하세요.
답변은 핵심 값만 간결하게 답하세요.
```

## Step 2: Advanced RAG (Hybrid Search + Re-ranking, 년도 필터링 없음)

### 파이프라인 구성

```
질문 → FAISS (Top-5) + BM25 (Top-5) → 합쳐서 최대 10개
     → Re-ranking (상위 3개) → parent_content 컨텍스트 구성 → LLM → 답변
```

### 설정값

| 항목 | 설정값 |
|------|--------|
| BM25 Retriever k | 5 |
| Vector Retriever k | 5 |
| Ensemble 방식 | RRF (Reciprocal Rank Fusion) |
| RRF 가중치 (vector : BM25) | 0.5 : 0.5 |
| Re-ranker 모델 | BAAI/bge-reranker-v2-m3 |
| Re-ranking 후 최종 Top-K | 3 |
| 메타데이터 필터링 | 없음 |

**주요 설계 결정**:
- Re-ranker 입력을 child chunk(300자) 대신 **parent_content(1500자)** 기준으로 스코어링 → 짧은 chunk로 인한 컨텍스트 손실 방지
- 년도 필터링 미적용 → Re-ranking이 연도 혼선을 자연스럽게 보정하는지 검증
- BM25는 매 실행 시 메모리에서 재구성 (FAISS는 지속 활용)

## 전체 인사이트

1. **청킹 전략이 파이프라인보다 우선**: 검색 알고리즘(Hybrid, Re-ranking, 필터링)을 모두 조합해도 45%였던 정답률이 청크 크기 변경 하나로 70%까지 올랐다. 의료급여 문서처럼 HTML 표 중심 문서에서는 청크가 표 행 단위를 온전히 담아야 LLM이 항목명과 수치를 함께 확인할 수 있다.

2. **Hybrid Search는 Re-ranking과 반드시 함께**: BM25가 2026 문서 내 '2025년 변경사항' 섹션 키워드에 과도하게 반응하여 Hybrid Only(30%)는 Basic RAG(40%)보다 낮았다. Re-ranking을 추가하면 쿼리-문서 관련성 재평가로 BM25 부작용이 보정되어 45%로 회복됐다.

3. **Re-ranking은 hard 문항 개선에 효과적**: Basic RAG에서 0/4였던 hard 문항이 Hybrid + Re-ranking에서 2/4로 개선됐다. 벡터 검색 단독으로 놓쳤던 청크를 BM25 키워드 매칭으로 확보하고, Re-ranking이 관련성 높은 청크를 상위로 올린 결과다.

4. **medium 난이도에서 청크 크기 효과 집중**: easy는 단순 수치 조회라 c300에서도 충분히 검색됐고, hard는 정보가 여러 청크에 분산되어 크기를 늘려도 한계가 있다. 표 구조가 핵심인 medium(33% → 83%)에서 개선 폭이 가장 컸다.

5. **연도 혼선 제거가 정답률 향상을 보장하지 않는다**: year_filter는 연도 혼선을 0%로 줄였지만, 필터링으로 후보 풀이 축소되면서 관련 청크가 탈락하는 부작용이 발생해 정답률이 45% → 30%로 오히려 하락했다. 연도 혼선 제거와 정답률 향상은 별개의 문제이며, 필터링 자체가 정답률 하락의 직접 원인이 될 수 있다.

6. **임베딩 포화는 필터링·Re-ranking으로 해결 불가**: 임베딩 포화(두 연도의 문서 내용이 거의 동일할 때 벡터 공간에서 두 청크가 같은 위치에 수렴해 연도 구분 자체가 불가능해지는 현상)는 2025·2026 문서가 동일 정책 내용을 반복 수록할 때 발생하며, 청크 내에 연도 정보를 텍스트로 명시하거나 연도별 별도 인덱스를 구성하는 방식이 근본적인 해결책이 될 수 있다.
