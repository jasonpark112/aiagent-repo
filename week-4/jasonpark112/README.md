# 4주차 과제: Basic RAG + Advanced RAG (Hybrid Search & Re-ranking)

## 1. 실행 환경 및 사용 모델

| 항목 | 내용 |
|------|------|
| 언어 | Python 3.14 |
| 프레임워크 | LangChain |
| LLM | gpt-4o-mini |
| 임베딩 모델 | text-embedding-3-small |
| Re-ranker | BAAI/bge-reranker-v2-m3 (로컬, sentence-transformers) |
| 벡터 저장소 | FAISS |
| Hybrid Search | FAISS (Dense Vector) + BM25Retriever (Sparse, langchain-community) |

---

## 2. 인덱싱 전략

2025년, 2026년 PDF를 각각 마크다운으로 변환한 후 아래 전략으로 청킹했다.

- **Parent chunk**: 1,500자 (overlap 100자) — LLM 컨텍스트용
- **Child chunk**: 300자 (overlap 50자) — 검색 정밀도용

각 child chunk의 메타데이터에 `source_year`, `parent_id`, `parent_content`를 저장하여 검색 시 child로 찾고, LLM에는 parent_content를 전달하는 Parent-Child 구조를 채택했다.

```python
metadata = {
    "source_year": "2025",       # 년도 필터링 기준
    "parent_id": "2025_p0",      # 중복 parent 제거용
    "parent_content": "...",     # LLM에 전달할 넓은 컨텍스트
}
```

## 3. Golden Dataset 설계

| 구분 | 문항 수 | source_year |
|------|---------|-------------|
| 2025년 전용 (easy/medium/hard) | 8문항 | 2025 |
| 2026년 전용 (easy/medium/hard) | 8문항 | 2026 |
| cross-year (2025 기준) | 2문항 | 2025+2026 |
| cross-year (2026 기준) | 2문항 | 2025+2026 |

**cross-year 설계 의도**: CT 검사 본인부담금(2025년 500원 → 2026년 5%)과 항정신병 장기지속형 주사제 본인부담률(2025년 5% → 2026년 2%)처럼 연도 간 수치가 바뀐 항목을 쌍으로 구성했다. 동일 질문 구조에서 년도만 달리하여 RAG 시스템의 년도 구분 능력을 직접 평가할 수 있다.

## Step 1: Basic RAG

### 파이프라인 구성

```
질문 → FAISS 벡터 검색 (Top-3) → parent_content 컨텍스트 구성 → LLM → 답변
```

- **Retriever**: `vectorstore.similarity_search(question, k=3)`
- **LLM**: `gpt-4o-mini`, temperature=0
- **프롬프트**: 컨텍스트에 `[출처 년도: YYYY]`를 명시하여 LLM이 년도를 인식하도록 구성

```
아래 컨텍스트를 바탕으로 질문에 답하세요.
각 컨텍스트에는 출처 년도가 표시되어 있습니다. 질문이 특정 년도를 묻는 경우 해당 년도의 정보만 사용하세요.
컨텍스트에 없는 내용은 "정보를 찾을 수 없습니다"라고 답하세요.
답변은 핵심 값만 간결하게 답하세요.
```

## Step 2: Advanced RAG (Hybrid Search + Re-ranking)

### 파이프라인 구성

```
질문 → FAISS (Top-5) + BM25 (Top-5) → 합쳐서 최대 10개
     → 년도 pre-filtering → Re-ranking (상위 3개) → parent_content 컨텍스트 구성 → LLM → 답변
```

### 설정값

| 항목 | 설정값 |
|------|--------|
| BM25 Retriever k | 5 |
| Vector Retriever k | 5 |
| Ensemble 가중치 (vector : BM25) | 0.5 : 0.5 |
| Re-ranker 모델 | BAAI/bge-reranker-v2-m3 |
| Re-ranking 후 최종 Top-K | 3 |
| 메타데이터 필터링 | 적용 (cross-year 제외) |

**주요 설계 결정**:
- Re-ranker 입력을 child chunk(300자) 대신 **parent_content(1500자)** 기준으로 스코어링 → 짧은 chunk로 인한 컨텍스트 손실 방지
- hybrid search 후 **년도 pre-filtering** 적용 → wrong year chunk가 re-ranking 후보에 포함되는 문제 방지
- BM25는 매 실행 시 메모리에서 재구성 (FAISS는 지속 활용)

### 인사이트

1. 가장 높은 정답률은 Hybrid + Re-ranking (no filter) 45%이다. 년도 필터를 제거하자 year_filter 버전(30%) 대비 15%p 상승했다. 필터가 후보 풀을 지나치게 좁혀 관련 청크를 탈락시키는 역효과가 있었다(q03, q14, q20 악화).

2. 년도 필터는 년도 정확도를 100%로 올리지만 정답률은 오히려 낮춘다. 두 연도 간 수치가 동일한 문항(q04, q14)에서는 필터 없이 반대 연도 청크를 가져와도 정답이 되기 때문에 필터의 이점이 제한적이다.

3. Hybrid Search 단독은 오히려 Basic RAG보다 년도 정확도가 낮다(51.4% vs 59.7%). BM25가 키워드 매칭으로 두 연도의 유사 문서를 동시에 끌어오면서 년도 혼동이 더 심해졌다.

4. hard 난이도는 4가지 방식 모두 0~1/4 수준이다. 복합 계산(q07)과 특수 조건(q08, q18)은 검색 방식 변경보다 청크 설계 개선이 필요한 공통 문제이다.

5. Re-ranking 자체의 효과는 긍정적이다. Hybrid Only(30%) 대비 no_filter(45%)는 동일 후보에서 Re-ranking만 추가했을 때 15%p 상승으로 Re-ranker의 기여가 유효하다.
